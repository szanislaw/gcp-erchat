Core Pipeline Specification

1. Input Validation & Boundary Defense

Located inside app/input_validator.py. Rejects threats before touching internal resources:

  • Length Checks: Blocks input strings outside of the 2-2000 character boundaries.
  • XSS Scans: Leverages 9 target regex vectors checking for malicious payloads (e.g., <script>, javascript:, inline handlers).
  • In-Text SQLi Protection: Evaluates 12 patterns checking for query breaks embedded inside natural phrasing text (e.g., ; DROP, UNION SELECT).

2. Rate Limiting

Located inside app/rate_limiter.py. Implements a token bucket algorithm restricting workloads to a hard threshold of 2 requests per second with a burst tolerance up to 10. Exceeded requests immediately fail fast with a HTTP 429 Too Many Requests status code.

3. Contextual Prompt Assembly

The framework assembles structural schemas formatted explicitly as SQL DDL definitions matching the underlying model's core training patterns:

  • DDL Serialization: Translates cached active schemas directly to standard CREATE TABLE syntax strings.
  • ENUM Guards: Loads DISTINCT field constraints into the prompt context block to eliminate hallucinated filters.
  • Access Control Constraints: Injects necessary WHERE property_uuid IN (...) requirements automatically into the workspace constraints text.

4. Execution & GPU Memory Config

Runs inside app/sqlcoder.py leveraging standard greedy decoding structures (do_sample=False, num_beams=1). Shipped defaults execute via float16 operations pulling roughly ~13 GB of VRAM. If resources are restricted, toggling USE_QUANTIZATION=true converts calculations into a compressed 4-bit NF4 structure reducing overhead to ~5 GB.

LRU Cache Architecture: A 500-entry memory matrix indexes calculations utilizing MD5 hashes derived from combined prompt configurations. Cache matches execute inside 0ms by routing directly around the GPU processor.

SQL Post-Processing Engine

Fixes are intentionally split across sequential execution boundaries to enforce isolation parameters:

Fixer Function Pipeline Stage Core Structural Purpose
fix_date_parse_to_to_date Stage A (sqlcoder.py) Transforms Athena-styled date_parse commands into valid Redshift TO_DATE() syntaxes.
fix_date_add_to_dateadd Stage A (sqlcoder.py) Converts unmapped ANSI/Presto date_add intervals over to DATEADD structures.
fix_table_names Stage B (main.py) Resolves hallucinated or malformed plural strings into verified whitelist physical target relations.
fix_spurious_department_join Stage B (main.py) Prunes unnecessary tables and hanging dependencies generated from model logic fallouts.
inject_property_filter Stage B (main.py) Acts as a critical fallback forcing secure multitenancy parameters into missing query frames.

Automated Self-Correction Architecture

When database operations encounter a RuntimeError layout, an integrated error handler initializes an automatic context correction cycle:

sequenceDiagram participant M as app/main.py participant RS as AWS Redshift participant SC as app/sqlcoder.py M->>RS: Submit Generated SQL Query (Attempt 0) RS-->>M: Return SQL Execution Error Code Context M->>M: Build Contextual Error Resolution Prompt M->>SC: Route Correction Query to GPU Engine SC-->>M: Return Reparsed SQL Draft String M->>RS: Submit Corrected SQL Query (Attempt 1) Note over M,RS: Process scales up to a maximum limit of 2 total correction retries.
Connection Allocation Overhead: Each distinct calculation request queries a fresh individual password-authenticated lifecycle sequence. This deliberate state isolation setup brings an expected ~50-100ms environment connection latency.
