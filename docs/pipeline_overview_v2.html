How It Works

A natural language question travels through eleven stages before results reach the client. Each stage is independently optimised for correctness and safety.

NLQ Input ──> [1. Validate & Sanitize] ──> [2. Rate Limiting] ──> [3. Target Resolution] │ └──> [4. Prompt Engineering] (DDL + ENUMs + Constraints) │ └──> [5. SQLCoder-7B AI Inference] ──> (LRU Cache Check) │ └──> [6. 17 SQL Fixers Post-Processing] │ └──> [7. Security Validation Layer] │ └──> [8. AWS Redshift Execution] (Self-Correct Loop) │ └──> [9. Display Detection & Formatting] ──> JSON Response
1 Input Validation & Sanitization
app/input_validator.py
Every question is checked for XSS payloads, SQL injection patterns, and length bounds (2–2000 characters) before touching any internal system. Dangerous content is rejected with HTTP 400 in strict mode. The sanitized version of the text is used for all downstream steps.
2 Rate Limiting
app/rate_limiter.py
A token bucket enforces 2 requests per second with a burst allowance of 10. Exceeding the limit returns HTTP 429 with a Retry-After header. This protects the GPU from being overwhelmed during multi-second inference.
3 Target Resolution
app/redshift_config.py
The redshift_target field in the request selects a database domain — either default (maintenance orders) or incident (hotel incident recovery). Each target has its own schema, table list, and access rules.
4 Prompt Engineering
app/prompt.py · app/schema_loader.py · app/query_normalizer.py

The system builds a rich prompt that provides the AI model with everything required to produce an accurate Redshift query:

  • Database Schema: DDL statements loaded directly from the Redshift metadata (cached).
  • Allowed Values: DISTINCT ENUMs injected to prevent hallucinated states.
  • Entity Hints: Pre-extracted names, locations, and explicit static calendar bounds.
  • Access Control Constraints: Compulsory tenancy structures are embedded directly.
5 AI Model Inference (GPU)
app/sqlcoder.py · defog/sqlcoder-7b-2
The prompt is sent to defog/sqlcoder-7b-2, a 7-billion parameter language model fine-tuned specifically for SQL generation. The model runs on GPU (float16, ~13 GB VRAM) and produces a SQL query in 2–10 seconds. Results are cached by prompt hash, so identical questions return instantly.
6 SQL Post-Processing — 17 Fixers
app/sqlcoder.py (Stage A) · app/main.py (Stage B)
Even the best SQL models make systematic mistakes. A two-stage post-processing pipeline applies 17 targeted regex-based fixers to correct known failure patterns (such as fixing dialect syntax like rewriting date_add to DATEADD or dropping missing keys/joins) before execution.
7 SQL Safety Validation
app/security.py
Before any SQL reaches the database, a security validator checks that no write operations (DROP, DELETE, INSERT, UPDATE, etc.) appear in the query, and that all referenced tables belong to an approved target whitelist.
8 Redshift Execution & Self-Correction Loop
app/redshift_client.py
The validated SQL is executed against Amazon Redshift. Results are capped at 100 rows. If an execution runtime error occurs, the system passes the database message context back into an internal Self-Correction Loop to retry generation up to twice automatically.
9 Display Type Detection & Formatting
app/display_hint.py · app/chart_formatter.py
The API evaluates the structured output and maps columns automatically into Chart.js friendly payload packages. It automatically suggests whether the dataset looks best as a metric, bar, line, or plain table display layout.

Analytics Domains

📦 Maintenance Orders (target: "default")

Tracks hotel maintenance work orders from creation through completion. Includes status, priority, location, department, and date dimensions.

Tables: maintenance_order, master_maintenance_status, master_job_priority, property_location
📦 Incident Recovery (target: "incident")

Tracks guest incident reports and recovery actions across hotel properties. Uses a pre-joined flat view — no SQL JOINs required.

View: mv_recovery_all (Fully denormalized view query)

Query & Display Types

Display Type Description Example Natural Language Question
▣ Metric Single numeric KPI — counts, totals, averages, percentages. "What is the average actual cost per incident?"
▬ Bar Chart Categorical comparisons — counts or averages grouped by a dimension. "Show maintenance order count by status"
～ Line Chart Time series trends — daily, weekly, or monthly aggregations. "Show the monthly trend of maintenance orders created"
≡ Table Raw record listings — recent items, multi-column detail views. "Show the 10 most recent incidents"

Performance Characteristics

  • Prompt Construction: ~5ms (schema info cached in-memory)
  • GPU Model Inference: 2–10 seconds (uncached cold run)
  • LRU Cache Read: 0ms (instant hash index lookup matches)
  • Redshift Query: ~200ms (direct standalone roundtrips)
